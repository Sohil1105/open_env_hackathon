Bug Report

Critical Bugs (P0 — Fix Before Any Deployment)

BUG-001 Dual consistency bonus — two diverging implementations
CriticalML / Grader
File: environment/rewards.py:37–55 vs environment/graders.py:185–247
compute_reward() recalculates its own consistency bonus with different logic and overwrites grading_result.consistency_bonus set by grade_action(). For High + Conditional Approve + 14%+, grade_consistency() returns +0.10 but rewards.py returns 0.0 — two functions disagree on the same decision. The authoritative source is ambiguous; callers see different scores depending on path taken.
BUG-002 global_session referenced 200 lines before its definition
CriticalBackend
File: server/app.py:261 (referenced) vs app.py:463 (defined)
The /reset handler body references global_session at line 261, but LifecycleSession() is instantiated at line 463. Works at runtime because Python function bodies are lazy, but any refactor or early startup call will raise a NameError crash.
BUG-003 monthly_expenses fallback uses existing_debt/12 — semantically wrong
CriticalBackend
File: server/app.py:660–661
When the UI submits monthly_expenses=0 (typical), the backend computes monthly_expenses = existing_debt / 12. Monthly expenses and debt repayment are entirely different things. This corrupts the ApplicantProfile passed to calculate_dynamic_ground_truth(), potentially shifting risk classifications.
BUG-004 LLM calls have no timeout — server can hang indefinitely
CriticalBackend
File: server/app.py:105–122
call_llm() calls client.chat.completions.create() with no timeout parameter. A slow or unresponsive HuggingFace router will block the /evaluate endpoint worker indefinitely (×5 sequential calls per request), eventually exhausting uvicorn's thread pool under any load.
BUG-005 fetch('/evaluate') does not check response.ok — silent failure
CriticalUI
File: static/index.html:1325–1330
executeStep() calls fetch('/evaluate') but never checks response.ok. A 422 (missing field), 500 (LLM error), or 401 (bad token) response is silently passed to displayResult(), which shows all decision fields as — with no error message visible to the user.
BUG-006 ApplicantInput.credit_score has no range validation — triggers 500 not 422
CriticalBackend
File: server/app.py:177 (class ApplicantInput)
ApplicantInput declares credit_score: int with no ge=300, le=850 constraint. An out-of-range value (e.g. 999) passes Pydantic validation at the route layer, then raises a ValidationError inside evaluate_applicant() when ApplicantProfile is constructed — returning an unhandled 500 instead of a clean 422 Unprocessable Entity.

Moderate Bugs (P1 — Fix Before Beta)

BUG-007 JavaScript validates only name and income — 8+ other fields unguarded
ModerateUI
File: static/index.html:1116–1157
changeWizardStep() only guards applicantName (step 1) and annualIncome (step 2). Credit score, loan amount, existing debt, and tenure all accept empty or negative values and are submitted to the backend. The HTML min/max on creditScore can be bypassed via browser devtools.
BUG-008 Missing domain fields: Purpose, Public Records, Inquiries
ModerateUI + Backend
File: static/index.html (UI form) + server/app.py (ApplicantInput)
Standard FICO underwriting uses Loan Purpose, Public Records count, and Credit Inquiries as risk signals. Neither the UI form nor the ApplicantInput schema includes these fields. The LLM receives an incomplete applicant profile, limiting decision accuracy on edge cases.
BUG-009 inference.py prints "TASK i/3" for an 8-task TASK_ORDER
ModerateInference
File: inference.py:255
print(f"TASK {i}/3: {task_id}") hardcodes the total as 3 but TASK_ORDER has 8 entries. A full inference run prints confusing output like "TASK 6/3", "TASK 7/3".
BUG-010 STAGE_PROMPTS dictionary defined but never used — dead code
ModerateBackend
File: server/app.py:450–454
A 400-character dict comprehension assigns identical prompt templates to all 8 task IDs, but evaluate_applicant() builds all prompts inline and never reads STAGE_PROMPTS. Dead code that implies a stage-routing capability that does not exist.
BUG-011 Grader docstrings say "Exact match: 1.0" but code returns 0.99
ModerateGrader
File: environment/graders.py:99, 130, 161
grade_risk_level(), grade_loan_decision(), and grade_interest_rate() all document "Exact match: 1.0" but return 0.99 for a perfect match. Any external code checking score == 1.0 will silently never match.
BUG-012 All task profiles omit documents_submitted — LLM sees "Missing critical files" for every task
ModerateEnvironment
File: environment/tasks.py:65–90 (and all other task profiles)
All 8 predefined ApplicantProfile instances omit documents_submitted, defaulting to []. get_underwriting_explanation() then renders "Pending / Missing critical files" in the ground truth explanation for every task — misleading the LLM about document completeness regardless of the scenario.

Info / Security

BUG-013 CORS allow_origins=["*"] — security risk in production
InfoBackend
File: server/app.py:141–147
CORSMiddleware configured with allow_origins=['*'] and allow_credentials=True. The CORS spec disallows credentials with wildcard origins in browsers. Acceptable for hackathon; must be restricted to known origins before production.
BUG-014 inference.py docstring says "3 tasks" — stale since TASK_ORDER expanded to 8
InfoInference
File: inference.py:7
The file-level docstring reads "Iterates through all 3 tasks (easy → medium → hard)". TASK_ORDER now has 8 tasks. Misleading for contributors and CI validators.
